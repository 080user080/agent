# Проєкт: Асистент МАРК (Voice/Text Assistant with Agent Capabilities)

## 1. Загальний опис

**МАРК** — локальний голосовий і текстовий асистент для Windows на Python.
Поточна мета проєкту — не просто голосовий помічник, а **агент**, який:
- отримує задачу українською мовою;
- будує план дій;
- виконує кроки через інструменти;
- перевіряє результат;
- пробує виправити невдалий крок;
- зберігає контекст виконання.

Основний стабільний сценарій зараз: **текстова взаємодія через GUI**.

## 2. Поточний стан проєкту (17.04.2026)

### ✅ Уже реалізовано

- GUI на Tkinter для діалогу з асистентом.
- Текстовий режим як основний робочий режим.
- STT і TTS інтегровані.
- Реєстр функцій `aaa_*.py`.
- Кеш, dispatcher і LLM-виклики.
- Стрімінг відповіді LLM у GUI.
- Довготривала пам'ять через `functions/core_memory.py`.
- Planner інтегрований у `main.py` і `logic_commands.py`.
- Асинхронне виконання плану через `TaskExecutor`.

### ✅ Оновлення цього етапу

Після останнього оновлення planner більше не є лише "чернеткою JSON-плану".

Тепер він має базовий агентний цикл:

`Goal -> Plan -> Act -> Verify -> Repair -> Save Context`

Що вже зроблено:
- planner сам вирішує, коли задача схожа на багатокрокову;
- список доступних інструментів береться з реєстру функцій, а не хардкодиться вручну;
- перед виконанням є базова перевірка безпеки плану;
- кожен крок проходить підготовку через execution context;
- після кожного кроку є валідація результату;
- якщо крок неуспішний, planner пробує згенерувати один repair-крок;
- контекст виконання оновлюється після кожного кроку;
- результати плану зберігаються в довготривалу пам'ять.
- додано сумісний `tool runtime`, який нормалізує результати інструментів у structured format без ламання GUI.

## 3. Що ще не дороблено

- Перевірка безпеки поки що **базова**, не повноцінна політика безпечного виконання.
- Repair-логіка вже є, але ще не багатоетапна.
- Коротка історія діалогу в `VoiceAssistant` усе ще обрізається до 10 повідомлень.
- Planner ще не робить повноцінне "перепланування всього дерева", лише repair одного кроку.
- Частина навичок повертає неструктурований текст, а для сильного агента краще мати єдиний формат результатів.

## 4. Актуальний статус

**Статус проєкту:** функціональний локальний асистент з уже інтегрованим базовим агентним циклом.

Це вже не просто GUI-асистент із набором окремих команд.
У проєкті є:
- planner,
- streaming,
- memory,
- async execution,
- step validation,
- repair attempt.

Правильне формулювання статусу на сьогодні:

**МАРК — це локальний асистент, який уже має базову агентну архітектуру, але ще потребує стабілізації, уніфікації інструментів і посилення безпеки.**

## 5. План розвитку до рівня "агент як Codex"

### Етап A. Minimum Viable Agent

- [x] Додати planner.
- [x] Підключити planner до реального виконання.
- [x] Додати базову пам'ять.
- [x] Додати валідацію кроків.
- [x] Додати хоча б один repair-крок після помилки.

### Етап B. Strong Agent Loop

- [x] Додати базове перепланування після невдалого repair-кроку.
- [ ] Додати структурований формат результату для всіх інструментів.
- [ ] Зберігати артефакти кроків: файли, шляхи, вивід, помилки, змінні.
- [ ] Навчити planner використовувати попередні результати між кроками без крихких текстових евристик.

### Етап C. Tool Runtime

- [x] Створити базовий runtime для інструментів із єдиним форматом результату.
- [x] Почати уніфікацію `create_file`, `edit_file`, `execute_python`, `open_program` за форматом відповіді.
- [x] Додати базові статуси типу `success / error / needs_confirmation / retryable`.
- [ ] Розвести "безпечні", "умовно небезпечні" і "заборонені" інструменти на рівні runtime.

### Етап D. Memory & Context

- [ ] Розширити `MemoryManager` до пам'яті сесії й пам'яті задачі.
- [ ] Додати окреме збереження проміжних результатів кроків.
- [ ] Прибрати жорстке обрізання короткої історії до 10 повідомлень або зробити його адаптивним.
- [ ] Додати summary-memory для довгих сесій.

### Етап E. Safety

- [x] Посилити правила безпеки planner через централізовану runtime policy.
- [x] Підключити `confirm_action` у ризиковані дії planner execution flow.
- [ ] Додати політику відмови для небезпечних або двозначних планів.
- [ ] Додати аудит виконаних дій.

### Етап F. Code Agent Mode

- [ ] Додати інструменти читання файлів, пошуку по проєкту, редагування коду, запуску перевірок.
- [ ] Додати роботу з git.
- [ ] Дати агенту цикл: знайти проблему -> змінити код -> перевірити -> виправити.
- [ ] Винести окремий system prompt для режиму coding agent.

## 6. Найближчий технічний пріоритет

Найближча велика задача:
